# Release Safety Checklist

ใช้ก่อนปล่อยแอปทุกครั้ง โดยเฉพาะรอบที่แตะระบบล็อกอิน การแจ้งเตือน การนำทาง หรือหน้าผู้ดูแล

## 1. ตรวจก่อน build

- เปิดแอปได้ทั้ง iOS และ Android โดยไม่เด้งออกตั้งแต่หน้าแรก
- รันตรวจ error ในไฟล์ที่แก้ล่าสุดและไม่มี error ใหม่ค้างอยู่
- ถ้าแก้ navigation, auth, notifications หรือ storage ให้รัน type check เพิ่มทุกครั้ง
- ตรวจว่า environment ที่ใช้ปล่อยถูกต้อง เช่น Firebase project, App Check, Sentry, EAS profile

## 2. ล็อกอินและสิทธิ์การใช้งาน

- ล็อกอินผู้ใช้ทั่วไปแล้วเข้าเฉพาะหน้าที่ควรเข้าได้
- ล็อกอินพยาบาลแล้วเข้า flow ของพยาบาลได้ครบ
- ล็อกอินโรงพยาบาลแล้วเข้า flow ของโรงพยาบาลได้ครบ
- บัญชีที่ไม่ใช่ผู้ดูแลเข้า admin dashboard ไม่ได้
- ออกจากระบบแล้วกด back หรือเปิด deep link ซ้ำ ต้องไม่หลุดเข้า protected screen
- สลับบัญชีไปมาแล้วข้อมูลค้างจากบัญชีก่อนหน้าไม่ติดมาด้วย

## 3. Push notification

- กรณีแอปเปิดอยู่: แตะ push แล้วพาไปหน้าที่เกี่ยวข้องถูกต้อง
- กรณีแอปอยู่ background: แตะ push แล้วเข้าแอปพร้อมไปหน้าที่ถูกต้อง
- กรณีเปิดแอปจากสถานะปิด: แตะ push แล้วรอจน auth และ navigation พร้อมก่อนค่อยพาไปหน้าเป้าหมาย
- ถ้า payload ไม่ครบ เช่น ไม่มี jobId หรือ conversationId แอปต้องไม่เด้งและไม่พาไปหน้าผิด
- เปิดจาก push ซ้ำ ๆ รายการเดิมแล้วต้องไม่เกิดการนำทางซ้ำหลายรอบ

## 4. Deep link และการนำทาง

- deep link ไปหน้าที่ต้องล็อกอิน ต้องถูกกันไว้ถ้ายังไม่ผ่าน auth
- deep link ที่ส่งพารามิเตอร์ไม่ครบ ต้องไม่ crash
- route ที่จำกัดตาม role เช่น applicants, my posts, payment, settings ต้องยังถูกกันครบ
- แตะจากการแจ้งเตือนเข้า ChatRoom, Applicants, Documents, Verification, Reviews ได้เมื่อข้อมูลครบเท่านั้น

## 5. ข้อมูลในเครื่องและ session

- logout แล้วข้อมูลผู้ใช้เดิมต้องไม่ค้างในเครื่องเกินจำเป็น
- nearby alert หรือ cache ที่ยังเก็บไว้ต้องเป็นข้อมูลขั้นต่ำ ไม่ใช่ profile เต็ม
- ล็อกอินบัญชีใหม่แล้วไม่เห็นข้อมูลแจ้งเตือนหรือสิทธิ์ค้างจากบัญชีก่อน

## 6. Admin dashboard smoke test

- เปิดหน้า admin dashboard แล้วโหลดข้อมูลครบ ไม่มีการ์ดล้นหรือปุ่มกดยากบนมือถือ
- เปลี่ยนแท็บ ภาพรวม สมาชิก งาน ข้อความ สื่อสาร สิทธิพิเศษ ได้ครบ
- กดทางลัดในหน้า overview แล้วไปหน้าปลายทางถูกต้อง
- เปิด modal ผู้ใช้ งาน และโค้ดสิทธิพิเศษได้
- pull to refresh แล้วข้อมูลรีโหลดได้โดยไม่ error

## 7. การสังเกตความผิดปกติหลังปล่อย

- เช็ก Sentry หลังปล่อยว่ามี crash ใหม่จาก ErrorBoundary หรือ navigation หรือไม่
- เช็กจำนวน push เปิดอ่านผิดปกติ หรือกดแล้วไม่ไปหน้าที่ต้องการ
- เช็ก feedback หรือรายงานจากทีมแอดมินใน 24 ชั่วโมงแรก
